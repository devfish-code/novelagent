## 1. 产品概述

### 1.1 产品定位

NovelAgent是一个AI驱动的长篇小说自动生成CLI工具，专注于解决AI写长篇小说时产生的**一致性问题**（角色设定漂移、伏笔遗忘、时间线混乱等）。通过五阶段工作流（需求理解→世界构建→大纲规划→逐章生成→全书校验）和强制自动修复机制，实现**一键生成高质量长篇小说**。

### 1.2 项目背景

当前AI写作工具普遍存在以下问题：
- 长文本超出上下文窗口导致信息丢失
- 角色设定前后矛盾
- 伏笔埋设后遗忘回收
- 时间线和地理信息混乱
- 缺乏全局结构把控

NovelAgent通过结构化工作流和世界状态数据库，将小说视为"有状态的世界模拟器"来解决这些问题。

### 1.3 核心价值主张

1. **全自动生成**：从创意输入到完整小说，全程无需人工干预
2. **一致性保证**：九项校验机制确保世界观、角色、时间线的一致性
3. **强制修复**：发现问题自动修复，不交付错误内容
4. **程序员友好**：npx一键使用，CLI/TUI双模式，支持工作目录指定

## 2. 用户角色定义

### 2.1 网文作者

- **描述**：需要批量生产小说内容的职业或半职业写手
- **目标**：快速生成小说初稿，减少重复劳动
- **痛点**：日更压力大，手动维护设定容易出错
- **技术水平**：中高级，熟悉命令行工具
- **使用频率**：每周生成1-3部小说框架

### 2.2 写作爱好者

- **描述**：热爱阅读小说，希望创造自己的故事世界
- **目标**：构建完整的小世界，沉浸式体验创作过程
- **痛点**：有创意但缺乏写作技巧和经验
- **技术水平**：初级到中级
- **使用频率**：每月生成1-2部完整小说

### 2.3 程序员创作者

- **描述**：有编程背景，喜欢使用命令行工具的创作者
- **目标**：用自动化工具实现创意
- **痛点**：现有GUI工具不够灵活，无法集成到工作流
- **技术水平**：专家级
- **使用频率**：不定期，用于个人项目

### 3.2 F-001：项目初始化(init)

#### 功能描述
在当前目录或指定目录创建NovelAgent项目结构和配置文件。

#### 用户故事
作为用户，我希望运行init命令后自动生成配置文件模板，以便快速开始小说创作。

#### 详细规格
- 入口：`npx @zb2947244682/novelagent init [options]`
- 操作流程：
  1. 检查目标目录是否已存在配置文件
  2. 创建项目目录结构（如果不存在）
  3. 生成config.yaml配置文件模板
  4. 创建必要的子目录（logs/、world/、outline/、chapters/、exports/）
- 输入参数：
  - `--dir, -d`: 指定工作目录（默认当前目录）
  - `--force, -f`: 强制覆盖已存在的配置
- 输出结果：
  - 成功：显示配置文件路径和下一步操作提示
  - 失败：显示错误原因（如目录无写入权限）

#### 边界情况
- 目录已存在配置文件：提示用户并使用--force覆盖
- 目录无写入权限：报错并提示检查权限
- 路径包含非法字符：校验并给出清晰的错误提示

#### 验收标准
- [ ] AC-001: 运行init后生成完整的目录结构
- [ ] AC-002: 生成的config.yaml包含所有必要配置项
- [ ] AC-003: 支持--dir参数指定任意工作目录
- [ ] AC-004: 重复init时正确提示用户

### 3.4 F-003：生成小说框架(framework)

#### 功能描述
根据用户提供的创意描述，生成完整的小说框架，包括需求文档、世界观、角色设定和大纲。

#### 用户故事
作为用户，我希望输入一句话创意，系统自动生成完整的小说框架，包含详细的设定和三层大纲。

#### 详细规格
- 入口：`npx @zb2947244682/novelagent framework "创意描述" --name <项目名> [options]`
- 操作流程（五阶段前3阶段）：
  1. Phase 1 - 需求理解：分析创意描述，生成结构化需求
  2. Phase 2 - 世界构建：生成世界观、角色档案、地点档案、规则体系
  3. Phase 3 - 大纲规划：生成全书大纲、卷大纲、章大纲
- 输入参数：
  - `创意描述`: 必填，支持自然语言描述
  - `--name, -n`: 项目名（必填，决定目录名）
  - `--dir, -d`: 工作目录
  - `--volumes`: 卷数（覆盖配置默认值）
  - `--chapters-per-volume`: 每卷章数
  - `--words-per-chapter`: 每章字数
- 输出结果：
  - 创建项目目录 `<工作目录>/<项目名>/`
  - 生成 `requirements.md` 需求文档
  - 生成 `world/` 目录下的世界观文件
  - 生成 `outline/` 目录下的大纲文件
  - 生成 `project.json` 项目元数据

#### 边界情况
- 项目名已存在：提示用户，支持--force覆盖
- 创意描述过短（少于10字）：警告提示描述可能不够详细
- AI生成失败：重试3次，仍失败则报错并保留日志

#### 验收标准
- [ ] AC-001: 根据创意生成完整的需求文档
- [ ] AC-002: 生成至少5个核心角色的详细档案
- [ ] AC-003: 生成三层大纲（全书→卷→章）
- [ ] AC-004: 生成世界规则和时间线
- [ ] AC-005: 所有文件按规范目录结构存放
- [ ] AC-006: 生成过程记录完整日志

### 3.5 F-004：生成章节(chapters)

#### 功能描述
根据已生成的大纲，逐章生成小说正文，包含自动校验和修复。

#### 用户故事
作为用户，我希望系统根据大纲自动生成所有章节，并确保内容前后一致，发现问题自动修复。

#### 详细规格
- 入口：`npx @zb2947244682/novelagent chapters <项目名> [options]`
- 操作流程（Phase 4核心循环）：
  1. 加载项目配置和大纲
  2. 检查已生成章节，确定起始位置（断点续传）
  3. 对每章执行：
     a. 组装上下文（大纲、角色档案、前文摘要等）
     b. 写作Agent生成初稿
     c. 提取Agent扫描并更新世界状态
     d. 九项并行校验
     e. 如有问题，进入修复循环（最多N轮）
     f. 风格润色
     g. 保存章节并更新日志
  4. Phase 5全书校验（全部章节完成后）
- 输入参数：
  - `项目名`: 必填
  - `--dir, -d`: 工作目录
  - `--volume, -v`: 仅生成指定卷
  - `--range, -r`: 生成指定范围（如"5-10"表示第5到10章）
  - `--chapter, -c`: 仅生成指定章节（用于重生成）
  - `--force`: 强制重生成已存在的章节
- 输出结果：
  - 在 `chapters/` 目录生成章节文件
  - 更新 `project.json` 章节元数据
  - 在 `logs/` 目录生成详细日志
  - 生成 `report.md` 生成报告

#### 九项校验
1. 世界规则校验：是否违反已设定的世界规则
2. 时空校验：角色能否在该时间出现在该地点
3. 信息逻辑校验：角色是否使用了不该知道的信息
4. 角色行为校验：行为是否符合性格档案和当前动机
5. 能力校验：展示的能力是否在能力范围内
6. 物品/状态校验：使用的物品是否确实持有
7. 伏笔校验：计划埋设/回收的伏笔是否完成
8. 常识/背景校验：是否出现与世界背景不符的事物
9. 叙事逻辑校验：大纲功能是否完成，因果链是否清晰

#### 边界情况
- 大纲不存在：提示先运行framework命令
- 某章修复N轮仍不通过：记录错误，跳过继续下一章
- 网络中断：保存进度，支持断点续传
- API限流：自动退避重试

#### 验收标准
- [ ] AC-001: 支持断点续传，从上次中断处继续
- [ ] AC-002: 每章生成后进行九项校验
- [ ] AC-003: 校验失败时自动修复（最多N轮）
- [ ] AC-004: 修复N轮仍失败则记录错误并继续
- [ ] AC-005: 生成过程中显示进度（当前章/总章数）
- [ ] AC-006: 单线程顺序生成，保证前后关联
- [ ] AC-007: 每章完成后立即保存，避免丢失

## 4. 非功能性需求

### 4.1 性能要求

- 单章生成时间：视模型和篇幅，平均2-5分钟
- 上下文组装时间：<1秒
- 校验时间：<5秒
- 内存占用：峰值<500MB（考虑到大上下文注入）

### 4.2 可靠性要求

- 每章生成后立即持久化，避免丢失
- 网络失败自动重试（指数退避）
- 异常错误详细记录，便于调试
- 支持断点续传

### 4.3 可维护性要求

- Core-Shell架构，业务逻辑与框架分离
- 完善的日志系统
- 模块化设计，便于扩展校验规则

### 4.4 兼容性要求

- Node.js 20+
- 支持Windows/macOS/Linux
- 支持标准OpenAI API格式的所有服务商
